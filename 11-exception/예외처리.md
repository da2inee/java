# 예외 처리 (Exception Handling)

실행 중 **뭔가 잘못됐을 때** JVM이 던지는 **에러 객체**를 **잡아서 처리**하는 게 예외 처리예요.  
try-catch로 잡으면 프로그램이 그대로 죽지 않고, 대체 동작을 하거나 로그를 남길 수 있어요.

---

## 1. try / catch / finally 기본

### 구조

```java
try {
    // 예외가 날 수 있는 코드
} catch (예외타입 e) {
    // 예외가 났을 때 처리
} finally {
    // 예외 나든 안 나든 (보통) 한 번은 실행 (선택)
}
```

### 예시 1: 숫자 파싱

```java
public static void main(String[] args) {
    try {
        String input = "abc";
        int num = Integer.parseInt(input);   // "abc"는 숫자 아님 → NumberFormatException
        System.out.println("숫자: " + num);
    } catch (NumberFormatException e) {
        System.out.println("숫자가 아닙니다. 메시지: " + e.getMessage());
    } finally {
        System.out.println("-- 처리 끝 --");
    }
}
// 출력:
// 숫자가 아닙니다. 메시지: For input string: "abc"
// -- 처리 끝 --
```

- **try** 안에서 `Integer.parseInt("abc")`가 **NumberFormatException**을 던짐.
- **catch**가 그 예외를 받아서 메시지만 출력.
- **finally**는 예외가 나든 안 나든 실행됨.

### 예시 2: 배열 인덱스

```java
int[] arr = { 10, 20, 30 };

try {
    System.out.println(arr[0]);   // 10
    System.out.println(arr[5]);   // 인덱스 5 없음 → ArrayIndexOutOfBoundsException
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("잘못된 인덱스: " + e.getMessage());
}
```

### 예시 3: null 참조

```java
String str = null;

try {
    System.out.println(str.length());   // str이 null → NullPointerException
} catch (NullPointerException e) {
    System.out.println("null입니다.");
}
```

## 2. catch 여러 개 (예외 타입별 처리)

한 try에 **여러 catch**를 두면, **던져진 예외 타입에 맞는** catch가 실행돼요.

```java
try {
    int n = Integer.parseInt("123");   // 정상
    int[] a = { 1 };
    System.out.println(a[10]);        // ArrayIndexOutOfBoundsException
} catch (NumberFormatException e) {
    System.out.println("숫자 아님");
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("인덱스 범위 초과");
} catch (Exception e) {
    System.out.println("그 외 예외: " + e.getMessage());
}
```

- **Exception**은 모든 예외의 부모라서, 맨 마지막에 두면 "그 외 전부" 처리할 수 있어요.
- 순서는 **구체적인 예외 → 넓은 예외(Exception)** 로 두는 게 좋아요.

## 3. Checked Exception vs Unchecked Exception

### 표로 보기

| 구분 | Checked Exception | Unchecked Exception |
|------|------------------|---------------------|
| **상속** | `Exception` 직계 (RuntimeException 제외) | `RuntimeException` 상속 |
| **컴파일** | 반드시 **try-catch** 또는 **throws** 선언 | 선언 안 해도 컴파일 됨 |
| **예시** | IOException, SQLException, ClassNotFoundException | NullPointerException, IllegalArgumentException, NumberFormatException |

### Checked 예시 – 반드시 처리하거나 선언

```java
// Checked: IOException. 이걸 그냥 던지면 컴파일 에러.
public void readFile() throws IOException {
    Files.readAllLines(Path.of("file.txt"));  // IOException 가능
}

// 호출하는 쪽에서 try-catch
public static void main(String[] args) {
    try {
        new MyClass().readFile();
    } catch (IOException e) {
        System.out.println("파일 읽기 실패: " + e.getMessage());
    }
}
```

- **throws IOException** = "이 메소드는 IOException을 던질 수 있으니, 호출하는 쪽에서 처리해라."
- 호출하는 쪽에서 **try-catch** 하거나, 또 **throws** 로 넘겨야 해요.

### Unchecked 예시 – 선언 없어도 됨

```java
// Unchecked: 선언 안 해도 컴파일 됨
public int divide(int a, int b) {
    return a / b;   // b가 0이면 ArithmeticException (Unchecked)
}
```

- **ArithmeticException**, **NullPointerException**, **IllegalArgumentException** 등은 **RuntimeException** 계열이라 **throws** 안 써도 됨.

---

## 4. throws – 예외를 호출한 쪽으로 넘기기

메소드 안에서 **직접 catch하지 않고**, "나를 부른 쪽에서 처리해라" 하고 넘길 때 **throws** 를 씁니다.

```java
// 예외를 처리하지 않고, 선언만 함
public String readFirstLine(String path) throws IOException {
    return Files.readAllLines(Path.of(path)).get(0);
}

// 호출하는 쪽에서 처리
public static void main(String[] args) {
    try {
        String line = readFirstLine("data.txt");
        System.out.println(line);
    } catch (IOException e) {
        System.out.println("파일 오류: " + e.getMessage());
    }
}
```

---

## 5. 예외 다시 던지기 (rethrow)

catch 안에서 **로그만 남기고**, 같은 예외를 다시 던질 수 있어요.

```java
public void process() throws IOException {
    try {
        readFile();
    } catch (IOException e) {
        System.err.println("로그: " + e.getMessage());
        throw e;   // 같은 예외를 다시 던짐 → 호출한 쪽에서 처리
    }
}
```

## 6. try-catch를 "쓸 때 vs 안 쓸 때" 차이 예시

### 예시 1) 안 잡으면(try-catch 없음) → 예외가 전파되고, 현재 메서드 흐름은 중단

```java
public static void main(String[] args) {
    System.out.println("A");
    int n = Integer.parseInt("abc"); // NumberFormatException 발생
    System.out.println("B");         // 실행 안 됨
}
```

- `Integer.parseInt("abc")`에서 예외가 터지면, **현재 메서드의 남은 코드(B)는 실행되지 않아요.**
- `main`까지 올라와도 아무도 안 잡으면, 프로그램은 예외 메시지를 찍고 **종료**돼요.

### 예시 2) 잡으면(try-catch) → 흐름을 통제하고, 다음 로직을 계속 실행 가능

```java
public static void main(String[] args) {
    System.out.println("A");
    try {
        int n = Integer.parseInt("abc");
        System.out.println("B"); // try 내부에서 예외가 나면 여기까지 안 옴
    } catch (NumberFormatException e) {
        System.out.println("숫자 변환 실패. 기본값 0 사용");
    }
    System.out.println("C"); // catch 처리 후 계속 실행됨
}
```

- try-catch를 쓰면 예외가 나도 **프로그램을 “어떻게 계속 진행할지”를 코드로 결정**할 수 있어요.

### 예시 3) "잡아서 바꾸기(예외 변환)" → 호출자가 이해할 수 있는 예외로 의미를 올림

외부 연동/DB/IO 같은 하위 계층 예외를 그대로 올리면, 상위 계층이 “무슨 의미인지” 판단하기 어려워요.  
그래서 **도메인 관점 예외로 바꿔서 던지는 패턴**을 많이 씁니다.

```java
// 도메인 관점 예외(예: 동의 업무에서 거절/실패 의미를 명확히)
class ConsentClientException extends RuntimeException {
    public ConsentClientException(String message, Throwable cause) {
        super(message, cause);
    }
}

public boolean requestConsent(String userId) {
    try {
        // 외부 시스템 호출(예: HTTP). 여기서 네트워크 예외/타임아웃 등 발생 가능
        return callExternal(userId);
    } catch (RuntimeException e) {
        // "외부 호출 실패"라는 의미로 바꿔서 던짐(원인(cause)은 보존)
        throw new ConsentClientException("외부 동의 시스템 호출 실패", e);
    }
}
```

- 장점: 상위 계층은 `RuntimeException` 덩어리를 해석하는 대신, **"외부 동의 호출 실패"**라는 의미로 처리할 수 있어요.
- 주의: “미성년이라 동의 불가”처럼 **예상 가능한 업무상 거절**을 예외로 만들지/응답값으로 처리할지는 팀 기준이 필요해요.

### 예시 4) "복구(recover)"가 가능할 때는 예외를 잡고 대체 경로로 진행

```java
public String loadNickname(String userId) {
    try {
        return callExternalNickname(userId);
    } catch (RuntimeException e) {
        // 외부 시스템이 죽었을 때도 서비스는 계속 돌게 하고 싶다면 대체값 반환
        return "손님";
    }
}
```

- 이런 케이스는 try-catch의 장점이 커요. **서비스를 죽이지 않고 품질만 낮춰서 계속 제공**할 수 있으니까요.
- 다만 “복구”를 남발하면 장애가 숨을 수 있으니, 보통은 **로그/메트릭(카운트)** 같은 관측 가능성도 같이 챙겨요.

## 7. 면접용 정리

| 질문 | 답변 |
|------|------|
| try-catch-finally 역할? | **try** = 예외 나는 코드, **catch** = 예외 처리, **finally** = 정리(보통 한 번은 실행) |
| Checked vs Unchecked? | **Checked** = 반드시 try-catch 또는 throws. **Unchecked** = RuntimeException 계열, 선언 안 해도 됨. |
| throws? | "이 메소드는 이 예외를 던질 수 있으니, **호출한 쪽**에서 처리해라"는 선언. |

---

## 8. 자주 쓰는 예외 타입 (참고)

- **NullPointerException** – null 참조로 메소드/필드 접근
- **ArrayIndexOutOfBoundsException** – 배열 잘못된 인덱스
- **NumberFormatException** – 문자열을 숫자로 바꿀 수 없을 때 (parseInt 등)
- **IllegalArgumentException** – 인자가 잘못됐을 때 (직접 throw 할 때 많음)
- **IOException** – 입출력 오류 (Checked)

## 9. 자주 묻는 것 (FAQ)

### Q. try-catch만 예외처리라고 해?

**아니요.** 예외 처리에는 다음이 다 포함돼요.

- **try-catch / try-catch-finally** – 예외를 잡아서 처리
- **throws** – "나 말고 호출한 쪽에서 처리해라" 선언
- **throw** – 개발자가 직접 예외 던지기
- **try-with-resources** – 리소스 자동 닫기 + 예외 처리

---

### Q. try에 예외가 나면 catch로 잡고 실행은 끝나는 거지?

**아니요.** catch 실행 **후**에 try-catch **밖의 다음 코드**가 **이어서 실행**돼요. 프로그램이 거기서 끝나지 않아요.

- try 안에서 예외 나면 → 그 다음 try 코드는 안 돌고 → 맞는 catch로 점프 → catch 실행 → (finally 있으면 실행) → **밖으로 나가서 다음 줄 계속 실행**.

---

### Q. finally는 예외처리랑 상관없이 항상 실행돼?

**네.** 예외가 나든 안 나든, catch에 잡히든 안 잡히든, **finally는 (정상 흐름이라면) 한 번은 실행**돼요.

- try 정상 종료 → finally 실행
- try에서 예외 → catch 실행 → finally 실행
- try에서 예외 → catch에 안 잡힘 → **그래도 finally 먼저 실행** 후 예외가 위로 전파
- try나 catch에서 **return** 해도 → **return 직전에** finally 실행

예외: `System.exit()` 호출, JVM이 죽는 경우 등에서는 finally가 안 돌 수 있어요.

---

### Q. finally는 왜 필요해? try-catch 밖에 쓰면 되잖아.

**try-catch 밖**에 쓰면 **실행이 안 되는 경우**가 있어요.

- try에서 예외가 나는데 **그 타입을 잡는 catch가 없으면** → 예외가 **위로 전파**되면서 catch 다음 줄(밖 코드)까지 **안 내려감** → "밖에 쓴 정리 코드"가 **실행 안 됨**.
- try나 catch 안에서 **return** 하면 → 밖 코드 **실행 안 됨**.

**finally**는 return이나 예외 전파가 나도 **한 번은 실행**되게 보장해 주니까, "반드시 해야 하는 정리(파일 닫기, 연결 반환 등)"를 넣을 때 필요해요.

---

### Q. try에서 에러 나면 catch로 잡는다며?

**catch는 "그 예외 타입"만 잡아요.** 타입이 안 맞으면 그 catch는 실행 안 하고 예외가 **위로 전파**돼요.

- 예: try에서 **ArithmeticException** 나는데 catch는 **IOException**만 잡으면 → 안 잡힘 → 예외가 위로 감 → try-catch **밖에 쓴 코드는 실행 안 됨**.
- catch 안에서 **return** 하거나 **throw** 해도 → 밖 코드 실행 안 됨.

그래서 "반드시 한 번 실행"이 필요하면 **finally**를 쓰는 거예요.

---

### Q. ArithmeticException이 뭐고, "위로 전파된다"는 게 뭐야?

**ArithmeticException**  
- **산술 연산** 중에 나는 예외예요.  
- 대표적으로 **정수를 0으로 나눌 때** (`10 / 0`) 발생해요.  
- RuntimeException 계열이라 throws 안 써도 되고, try-catch로 잡을 수 있어요.

**위로 전파(propagate)**  
- 예외를 **여기서 안 잡으면**, 그 예외가 **이 메서드를 호출한 쪽**으로 넘어간다는 뜻이에요.  
- 호출 스택을 따라 한 단계씩 **위(호출한 쪽)**로 전달돼요.  
- 끝까지 아무도 안 잡으면 프로그램이 그 예외 메시지와 함께 종료해요.

---

이 정도만 익혀 두면 예외 처리 흐름과 면접 대답에 충분해요.
