# HTTP / 쿠키 / 세션

## HTTP가 “stateless(무상태)”라는 말
- HTTP 요청은 기본적으로 **이전 요청의 상태를 기억하지 않는다.**
- 그래서 서버 입장에서는 요청이 올 때마다 “이 사람이 누구지?”를 **매번 다시 식별**해야 한다.

## 그럼 “로그인 상태 유지”는 어떻게 해?
대표적으로 **쿠키**와 **세션**을 이용한다.

## 쿠키(Cookie)
- 브라우저가 저장하는 **작은 데이터**(키-값)
- 서버가 응답 헤더로 쿠키를 내려주면, 브라우저는 이후 요청에 **자동으로 쿠키를 포함**해 요청한다.

### 쿠키 옵션에서 중요한 것들
- **HttpOnly**: JS로 쿠키 접근을 막아 **XSS**에 강해짐
- **Secure**: HTTPS에서만 전송
- **SameSite**: 다른 사이트에서 요청할 때 쿠키 전송을 제어(**CSRF** 방어에 중요)

## 세션(Session)
- “로그인 상태” 같은 정보를 **서버가 저장**해두는 방식
- 보통 서버는 세션 저장소에 상태를 저장하고,
  - 클라이언트에는 “세션을 찾을 수 있는 키”만 준다 (예: `JSESSIONID`)
  - 이 키가 보통 **쿠키**로 저장되어 매 요청에 같이 간다

## 한 줄 비교 (세션 vs 토큰)
- **세션 기반**: 상태를 **서버**가 들고 있고, 클라이언트는 “세션 키”만 보냄
- **토큰 기반(JWT 등)**: 상태(클레임)를 **클라이언트가 들고** 요청마다 토큰으로 증명

## 한 줄 정리
HTTP는 무상태라서 로그인 같은 “상태 유지”가 필요하면 **쿠키/세션(또는 토큰)** 같은 별도 메커니즘을 붙인다.
