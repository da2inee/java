# JWT(JSON Web Token)

## JWT가 뭐야?
JWT는 **로그인한 사용자를 증명하는 “서명된 토큰”**이다.  
서버가 세션을 저장하지 않아도(상태를 덜 들고 있어도) 요청마다 “내가 누구인지”를 확인할 수 있게 해준다.

## JWT 형태
```
header.payload.signature
```

### 1) Header
- 토큰 타입, 서명 알고리즘 정보 (예: HS256)

### 2) Payload
- 사용자/권한/만료 같은 “클레임(Claims)”이 들어간다.
- 대표 클레임
  - `sub`(subject): 보통 사용자 식별자(예: userId)
  - `exp`: 만료 시간
  - `iat`: 발급 시간

**중요**: Payload는 “암호화”가 아니라 **Base64 인코딩**이라서 **누구나 내용을 볼 수 있다.**  
→ 주민번호/비밀번호 같은 민감정보를 넣으면 안 된다.

### 3) Signature
- `header + payload`를 서버의 키로 서명한 값
- 서버는 요청이 올 때 **서명 검증**으로 “위조 여부”를 확인한다.

## JWT 인증 흐름(웹)
1) 로그인 성공
2) 서버가 JWT 발급
3) 클라이언트가 저장
4) 이후 요청마다 보통 아래 헤더로 전송

```
Authorization: Bearer <access-token>
```

서버는 매 요청마다:
- 서명 검증
- `exp` 만료 확인
- `sub` 등으로 사용자 식별

## 장점 / 단점(트레이드오프)
- **장점**: 서버 상태(세션) 부담이 줄고, 분산 환경에서 확장하기 편함
- **단점**: 토큰이 탈취되면 위험하고, “강제 로그아웃(폐기)”이 까다로움(특히 access 토큰)

## 한 줄 정리
JWT는 **서명으로 위조를 막는 토큰**이고, payload는 숨겨지지 않으니(암호화 아님) 민감정보를 넣지 않는다.
