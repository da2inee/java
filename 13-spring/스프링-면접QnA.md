4. AOP를 왜 쓰나?
- 답:
  - 핵심: **횡단 관심사(Cross-cutting concern)** 를 핵심 비즈니스 로직과 분리해서 **중복을 줄이고 일관성을 보장**하기 위해 쓴다.
  - 예: 로깅, 트랜잭션, 성능 측정, 보안/권한 체크, 캐싱, 예외 공통 처리 등은 여러 계층/메서드에 반복되기 쉬운데 AOP로 한 곳에서 적용할 수 있다.
  - 장점:
    - 비즈니스 코드는 비즈니스에만 집중(SRP) → 코드 가독성/유지보수성 향상
    - 정책 변경(로그 포맷, 권한 규칙 등)을 **한 군데**에서 바꿔도 전체에 반영
    - 공통 로직 적용 누락/불일치(“어떤 메서드는 로그 있고 어떤 메서드는 없음”)를 줄임
  - 스프링에서의 대표 사례: `@Transactional`은 AOP 프록시로 메서드 호출 전/후에 트랜잭션 시작-커밋/롤백을 끼워 넣는다.
  - 주의: 프록시 기반이라 **self-invocation(같은 클래스 내부 호출)** 에서는 advice가 안 타는 경우가 있어 설계/분리(빈 분리)로 피한다.

5. @Transactional이 동작하는 방식?
- 답:
  - 스프링은 `@Transactional`이 붙은 메서드에 대해 **프록시(AOP)** 를 만들어 트랜잭션을 적용한다.
  - 메서드 호출 전 **트랜잭션 시작**, 정상 종료 시 **커밋**, 예외 발생 시 **롤백**한다.
  - 기본 롤백 규칙은 **RuntimeException/Error는 롤백**, **Checked Exception은 커밋**이며, 필요하면 `rollbackFor`로 조정한다.
  - 프록시 기반이라 **같은 클래스 내부의 자기 호출(self-invocation)** 은 트랜잭션이 적용되지 않을 수 있다는 점을 주의한다.
