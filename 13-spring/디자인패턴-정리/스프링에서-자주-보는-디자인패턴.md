# 스프링에서 자주 보는 디자인 패턴

## 디자인 패턴이란?
- 반복되는 설계 문제에 대한 검증된 해결 방식(설계 템플릿)이다.
- 코드를 그대로 복사하는 것이 아니라, 클래스/객체의 역할과 협력 구조를 재사용한다.
- 목표는 변경에 강하고(유지보수), 확장 가능한 코드를 만드는 것이다.

## 왜 배우는가?
- 코드 구조를 빠르게 이해할 수 있다.
- 기능 추가 시 기존 코드를 덜 깨고 확장할 수 있다.
- 팀 내에서 공통 설계 언어(예: "전략 패턴으로 분리하자")로 소통할 수 있다.

## 1) Singleton 패턴
### 개념
- 애플리케이션에서 인스턴스를 하나만 두고 공유하는 패턴.

### 스프링에서
- 스프링 빈의 기본 스코프는 singleton이다.
- 같은 빈을 여러 번 조회해도 동일 인스턴스를 반환한다.

### 장점/주의
- 장점: 메모리 절약, 공통 상태/기능 공유.
- 주의: 가변 상태를 빈에 두면 동시성 문제가 생길 수 있다.

## 2) Factory 패턴
### 개념
- 객체 생성 책임을 별도 컴포넌트로 분리하는 패턴.

### 스프링에서
- `BeanFactory`, `ApplicationContext`가 대표적.
- 개발자는 생성 과정보다 "무엇을 주입받아 쓸지"에 집중한다.

### 장점
- 생성 로직이 분리되어 결합도가 낮아진다.
- 구현체 변경이 쉬워진다.

## 3) Strategy 패턴
### 개념
- 알고리즘(동작)을 인터페이스로 캡슐화하고, 런타임에 교체 가능한 패턴.

### 스프링에서
- 인터페이스 + 구현체 빈 조합으로 쉽게 적용한다.
- 예: `DiscountPolicy`를 `FixDiscountPolicy`, `RateDiscountPolicy`로 교체.

### 장점
- if-else 분기를 줄이고 확장에 유리하다.
- 테스트 시 Mock/Stub 전략으로 대체하기 쉽다.

## 4) Template Method 패턴
### 개념
- 공통 처리 흐름(뼈대)은 상위 클래스가 정의하고, 세부 단계는 하위 클래스가 구현.

### 스프링에서
- `JdbcTemplate`의 이름 자체가 이 패턴 의도를 잘 보여준다.
- 공통 자원 처리(연결/예외/정리)는 템플릿이 담당, 핵심 로직만 콜백으로 분리.

### 장점
- 중복 코드 제거.
- 공통 규칙을 강제하면서도 일부 단계는 유연하게 확장 가능.

## 5) Proxy 패턴
### 개념
- 실제 객체에 접근하기 전, 대리 객체를 통해 부가기능을 제공하는 패턴.

### 스프링에서
- AOP, `@Transactional`, 보안/로깅 등에 프록시가 자주 쓰인다.
- 핵심 비즈니스 코드와 부가기능을 분리할 수 있다.

### 장점
- 관심사 분리(비즈니스 로직 vs 공통 부가기능).
- 코드 중복 감소.

## 6) Observer 패턴
### 개념
- 객체 상태 변화(이벤트)를 구독자들에게 알리는 패턴.

### 스프링에서
- `ApplicationEventPublisher`, `@EventListener`로 구현 가능.
- 주문 완료 후 알림 발송, 포인트 적립 등 후처리를 느슨하게 연결할 때 유용.

### 장점
- 컴포넌트 간 결합도를 낮춘다.
- 기능을 이벤트 구독으로 점진적으로 추가하기 좋다.

## 실무에서 어떻게 써야 하나?
- 패턴을 "외우기"보다 "문제 상황"에서 떠올리는 연습이 중요하다.
- 작은 단위(인터페이스 분리, 생성 책임 분리)부터 적용한다.
- 과도한 패턴 적용은 오히려 복잡도를 높일 수 있다.

## 빠른 선택 가이드
- 객체 생성이 복잡하다 -> Factory
- 동작을 교체해야 한다 -> Strategy
- 공통 흐름 + 일부 단계만 다르다 -> Template Method
- 부가기능을 분리하고 싶다 -> Proxy
- 후처리를 느슨하게 연결하고 싶다 -> Observer

## 학습 체크리스트
- [ ] 각 패턴의 목적을 1문장으로 설명할 수 있다.
- [ ] 스프링 기능(`@Transactional`, 이벤트, 빈 관리)과 연결해 설명할 수 있다.
- [ ] 내 코드에서 if-else 분기를 Strategy로 바꿀 수 있다.
- [ ] 공통 로직을 Template/Proxy 방식으로 분리해본 경험이 있다.
