# REST API 핵심 정리

## REST API란?
- REST(Representational State Transfer) 스타일을 따르는 API 설계 방식이다.
- 서버의 데이터(리소스)를 URL로 표현하고, HTTP 메서드로 동작을 구분한다.
- 보통 JSON 형식으로 데이터를 주고받는다.

## 핵심 원칙
## 1) 리소스 중심 URL
- 동작(verb)보다 대상(resource, noun)을 URL에 표현한다.
- 좋은 예: `/users`, `/orders/1`, `/products/10/reviews`
- 피할 예: `/getUsers`, `/createOrder`, `/deleteProduct`

## 2) HTTP 메서드 의미 일치
- `GET`: 조회
- `POST`: 생성
- `PUT`: 전체 수정(대체)
- `PATCH`: 부분 수정
- `DELETE`: 삭제

## 3) 무상태(Stateless)
- 각 요청은 독립적이어야 한다.
- 서버는 이전 요청의 클라이언트 상태를 세션처럼 의존하지 않는 방향이 좋다.

## 4) 일관된 응답 구조
- 성공/실패 응답 포맷을 팀 기준으로 통일한다.
- 예시(성공): `{"data": {...}, "message": "ok"}`
- 예시(실패): `{"errorCode": "USER_NOT_FOUND", "message": "..."}`

## 5) 표준 상태코드 사용
- `200 OK`: 조회/수정 성공
- `201 Created`: 생성 성공
- `204 No Content`: 삭제 성공(응답 본문 없음)
- `400 Bad Request`: 잘못된 요청(검증 실패 등)
- `401 Unauthorized`: 인증 필요
- `403 Forbidden`: 권한 없음
- `404 Not Found`: 리소스 없음
- `409 Conflict`: 중복/충돌
- `500 Internal Server Error`: 서버 내부 오류

## URL 설계 가이드
- 복수형 명사 사용: `/users`
- 계층 관계는 슬래시로 표현: `/users/{userId}/orders`
- 일관된 네이밍 유지(kebab-case 또는 snake_case 등 팀 규칙 통일)
- 버전 전략 고려: `/api/v1/users`
- 쿼리 파라미터는 필터/정렬/페이지네이션에 사용
  - 예: `/users?page=0&size=20&sort=createdAt,desc`

## 스프링부트 기본 매핑 예시
- `GET /api/users` -> 목록 조회
- `GET /api/users/{id}` -> 단건 조회
- `POST /api/users` -> 사용자 생성
- `PATCH /api/users/{id}` -> 사용자 부분 수정
- `DELETE /api/users/{id}` -> 사용자 삭제

## 요청/응답 DTO 분리 권장
- 엔티티를 API로 직접 노출하지 않는다.
- Request DTO, Response DTO를 분리해 API 계약을 안정적으로 유지한다.

## 예외 처리 권장
- 목표: "장애"와 "업무상 거절"을 구분해, 클라이언트/로그/알람/재시도 정책이 흔들리지 않게 만든다.

### 1) 분류 기준(반드시 팀에서 합의)
- **업무상 거절(예상 가능한 실패)**: 입력/상태가 정책에 맞지 않아 처리가 거절됨
  - 예: **미성년(19세 미만)이라 동의 불가**, 약관 미동의, 잔액 부족, 한도 초과, 권한 부족(정책상)
- **시스템 실패(장애/비정상)**: 네트워크/타임아웃/DB 장애/버그/외부 시스템 장애 등으로 처리 자체가 불가

> 핵심: "미성년이라 fail"은 대부분 **업무상 거절(정상 시나리오)**로 보는 게 운영/모니터링에 유리하다.

### 2) API에서 실패를 표현하는 2가지 방식
#### A. HTTP 상태코드로 표현(권장)
- **업무상 거절(예상 가능한 실패)**: 4xx + 표준 에러 바디
  - 권장 후보: `400`(요청 자체가 정책 위반), `403`(정책상 금지), `409`(상태 충돌), `422`(의미상 처리 불가)
  - 예: 미성년 동의 불가 → `422` + `UNDER_19` 같은 에러코드
- **시스템 실패(장애)**: `500` 계열(또는 게이트웨이/외부 연동이면 `502/503/504`) + 표준 에러 바디

#### B. 200 OK + 바디에 success=false로 표현(비권장, 하려면 규칙 강화)
- 장점: 클라이언트가 예외 없이 응답을 받음
- 단점: 모니터링/재시도/서킷브레이커가 HTTP 실패 기준이면 장애가 숨을 수 있음
- 이 방식을 쓰면 반드시:
  - **`success`/`resultCode`가 없으면 계약 위반**으로 간주
  - 장애와 거절을 구분할 수 있는 **상세 코드 체계**를 설계(예: `category=BUSINESS|SYSTEM`)

### 3) 권장 에러 응답 표준(예시)
- 실패 응답은 항상 다음 필드를 포함하도록 통일한다.
  - **errorCode**: 기계가 처리할 코드(고정/문서화)
  - **message**: 사람이 읽을 메시지(로깅/디버깅용)
  - **traceId**: 요청 추적용(로그 상관관계)
  - (선택) **details**: 필드 검증 오류 목록 등

예시:
- `{"errorCode":"UNDER_19","message":"미성년자는 동의할 수 없습니다.","traceId":"..."}`

### 4) 스프링에서의 예외 처리 패턴(서버)
- 컨트롤러 단에서 try-catch로 덕지덕지 처리하지 말고, 예외를 던지고 한 곳에서 매핑한다.
- **`@RestControllerAdvice` + `@ExceptionHandler`**로 공통 응답을 만든다.
- 예외를 2계층으로 나누는 걸 추천:
  - **BusinessException(예상 가능한 거절)**: 4xx로 매핑, 에러코드 고정
  - **SystemException(장애/비정상)**: 5xx로 매핑, 내부 상세는 숨기고 traceId로 추적

### 5) 외부 시스템 연동 시(클라이언트) 처리 원칙
- 외부 연동에서 가장 중요한 건 "우리 도메인"으로 정상화(normalize)하는 것
  - 외부가 4xx로 주든 200으로 주든, 우리 서비스 내부에선 **BusinessReject vs SystemFailure**로 분기
- 재시도/알람 원칙(권장)
  - **SystemFailure(타임아웃/5xx/네트워크)**: 재시도/서킷브레이커/알람 대상
  - **BusinessReject(UNDER_19 등)**: 재시도 X, 사용자에게 안내 메시지/코드 전달

### 6) 체크 포인트(팀 합의 항목)
- 미성년/약관 미동의 같은 케이스는 **업무 거절(4xx)**로 처리할지?
- 업무 거절의 대표 상태코드는 무엇으로 통일할지? (`400` vs `403` vs `422`)
- 에러코드 네이밍 규칙(예: `UNDER_19`, `CONSENT_REQUIRED`, `LIMIT_EXCEEDED`)
- 외부 연동 에러를 우리 에러코드로 매핑하는 테이블/문서 소유자는 누구인지?

## 페이징/정렬/검색
- 목록 API는 초기에 페이징을 고려한다.
- 응답에 메타 정보(현재 페이지, 총 개수 등)를 포함하면 프론트 연동이 편하다.

## 체크리스트
- [ ] URL이 리소스 중심으로 설계되어 있다.
- [ ] HTTP 메서드와 동작 의미가 맞다.
- [ ] 상태코드를 상황에 맞게 반환한다.
- [ ] 성공/실패 응답 포맷이 일관적이다.
- [ ] 공통 예외 처리(`@RestControllerAdvice`)가 적용되어 있다.
- [ ] **업무상 거절(예상 가능)**과 **시스템 실패(장애)**가 구분되어 있다.
- [ ] 외부 시스템 연동 오류가 우리 도메인 에러코드로 정상화되어 있다(매핑 규칙 문서화).
- [ ] DTO 기반으로 API 계약을 관리한다.
