### Day 1 빠른 메모 (핵심 한 줄)
- **IoC**: 객체 생성/관리 제어권이 개발자에서 스프링으로 넘어가는 것
- **DI**: IoC를 구현하는 대표 방식. `new` 대신 생성자/필드/세터로 주입받기
- **Bean**: 스프링이 생성하고 관리하는 객체

```java
// DI 예시: new A() 하지 않고, 생성자로 주입받음
public Service(A a) {
    this.a = a;
}
```

## Day 2 - MVC / REST

- [ ] 컨트롤러 어노테이션 정리
- [ ] 공통 예외 처리 기본 구현

### Day 2 빠른 메모 (MVC 핵심)
- **M**odel: 데이터/비즈니스 결과
- **V**iew: 사용자에게 보여줄 결과 (템플릿 또는 JSON)
- **C**ontroller: 요청 받고 서비스 호출 후 응답 반환

#### Spring MVC 요청 흐름
1. 클라이언트가 HTTP 요청 전송
2. `DispatcherServlet`이 요청 수신
3. `HandlerMapping`이 매핑된 컨트롤러 메서드 탐색
4. 컨트롤러 실행 (`@PathVariable`, `@RequestParam`, `@RequestBody` 바인딩)
5. 서비스 로직 호출 후 결과 반환
6. ViewResolver(뷰) 또는 JSON 응답으로 클라이언트에 전달

#### REST에서 자주 쓰는 어노테이션
- `@RestController`: 반환값을 응답 본문(JSON 등)으로 전달
- `@GetMapping`, `@PostMapping`, `@PatchMapping`, `@DeleteMapping`: HTTP 메서드별 매핑
- `@PathVariable`: URL 경로 변수 받기 (`/users/{id}`)
- `@RequestParam`: 쿼리 파라미터 받기 (`?name=kim`)
- `@RequestBody`: 요청 JSON 본문을 DTO로 받기

#### 면접 한 줄 답
- "Spring MVC는 DispatcherServlet 중심으로 요청을 컨트롤러에 매핑하고, 비즈니스 로직을 처리한 뒤 View 또는 JSON 응답을 반환하는 구조입니다."

## Day 3 - 트랜잭션 / DB

- [ ] N+1, 인덱스 기본 복습

### Day 3 빠른 메모 (트랜잭션 핵심)
- **트랜잭션**: 여러 DB 작업을 “하나의 작업 단위”로 묶어서 **전부 성공하면 Commit**, 하나라도 실패하면 **Rollback**하는 것
- **ACID**
  - **Atomicity(원자성)**: 전부 아니면 전무
  - **Consistency(일관성)**: 제약조건/규칙을 깨지 않는 상태로만 변함
  - **Isolation(격리성)**: 동시에 실행돼도 서로 간섭을 최소화
  - **Durability(지속성)**: 커밋된 결과는 장애가 나도 유지

#### 스프링에서 트랜잭션 경계 잡는 법
- 보통 **Service 계층 메서드**에 `@Transactional`을 붙여 “메서드 시작~끝”을 트랜잭션으로 잡는다.
- **정상 종료**: Commit
- **예외 발생**: Rollback(기본 규칙은 아래 참고)

#### 롤백 규칙(기본)
- **RuntimeException / Error** 발생 시: 기본적으로 **Rollback**
- **Checked Exception(Exception)** 발생 시: 기본적으로 **Commit**
- 체크 예외도 롤백시키고 싶으면:

```java
@Transactional(rollbackFor = Exception.class)
```

#### 전파(Propagation) = 트랜잭션 중첩 호출 규칙
- **REQUIRED(기본)**: 기존 트랜잭션이 있으면 합류, 없으면 새로 시작
- **REQUIRES_NEW**: 무조건 새 트랜잭션 시작(기존은 잠시 보류)
- **NESTED**: savepoint 기반(환경/DB에 따라 제약)

#### 격리 수준(Isolation) 한 줄 정리
- 동시에 실행되는 트랜잭션 때문에 생기는 문제(Dirty/Non-repeatable/Phantom)를 **어느 정도 허용/차단할지** 정하는 옵션
- 실무에서는 보통 **DB 기본값**(많이 쓰는 값: READ COMMITTED)을 우선 사용하고, 필요할 때만 조정한다.

#### 자주 나오는 주의점
- `@Transactional`은 보통 **프록시 기반**이라서, 같은 클래스 내부에서 자기 메서드를 직접 호출하면 트랜잭션이 안 걸릴 수 있다.
- 조회 전용이면:

```java
@Transactional(readOnly = true)
```
