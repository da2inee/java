# 스프링 (Spring) 정리

**스프링** = 자바로 웹·엔터프라이즈 앱을 만들 때 쓰는 **프레임워크**예요.  
객체 생성·의존 관계를 **스프링이 대신 관리**해 주고, 웹 요청 처리·DB 연동 등을 쉽게 해 줘요.

---

## 스프링 핵심 개념 정리 (면접용)

### IoC (Inversion of Control, 제어 역전)

- **뜻**: 원래는 "내가(new로) 객체를 만들고 제어"하는데, **스프링(컨테이너)이 객체를 만들고 관리**하게 되면서 "제어가 역전"된 것.
- **한 줄**: 객체 생성·생명 주기를 **개발자가 아니라 스프링**이 담당한다.
- **면접 답**: "IoC는 객체 생성과 관리를 프레임워크가 하도록 제어가 넘어가는 것이고, 스프링에서는 Bean 컨테이너가 그 역할을 합니다."

---

### DI (Dependency Injection, 의존성 주입)

- **뜻**: 클래스가 필요로 하는 **다른 객체(의존)**를, **코드 안에서 new 하지 않고**, **스프링이 밖에서 넣어 주는 것**.
- **예**: `OrderService`가 `OrderRepository`를 쓸 때  
  - 기존: `OrderRepository repo = new OrderRepository();`  
  - DI: 생성자/필드로 `OrderRepository`를 **받기만** 하고, 스프링이 **주입**해 줌.
- **왜 쓰나**  
  - **결합도 감소**: 구현체를 바꿀 때 생성하는 쪽 코드를 안 건드려도 됨.  
  - **테스트 용이**: 테스트 시 Mock 객체를 넣어 주기 쉬움.  
  - **관심사 분리**: "무엇을 쓰는지"는 정하고, "누가 만드는지"는 스프링이 담당.
- **면접 답**: "DI는 필요한 의존 객체를 외부에서 주입받는 방식으로, new를 하지 않아 결합도가 낮아지고 테스트와 교체가 쉽습니다."

---

### Bean (빈)

- **뜻**: 스프링이 **생성하고 관리하는 객체** 하나. "스프링이 관리한다" = 생성 시점, 스코프(범위), 다른 빈에 주입해 주는 것까지 스프링이 처리.
- **특징**  
  - 기본 스코프는 **싱글톤**: 같은 타입의 빈이 앱 안에 **하나**만 있음.  
  - 스프링이 **어디서, 언제** 만들지 정함.
- **등록 방법**  
  - **@Component, @Service, @Controller, @RestController, @Repository**  
    → 클래스에 붙이면 스프링이 스캔해서 **그 클래스의 인스턴스 하나**를 빈으로 등록.  
  - **@Bean**  
    → 설정 클래스(@Configuration)의 **메소드**에 붙이면, 그 메소드가 **반환하는 객체**가 빈으로 등록됨. (외부 라이브러리 객체 등)
- **면접 답**: "Bean은 스프링 컨테이너가 생성·관리하는 객체이고, 보통 싱글톤으로 하나만 두고, DI로 필요한 곳에 주입합니다."

---

### 생성자 주입을 쓰는 이유 (권장 방식)

- **방식**: 의존성을 **필드**나 **setter**가 아니라 **생성자 파라미터**로 받음.  
  `public OrderService(OrderRepository orderRepository) { this.orderRepository = orderRepository; }`
- **이점**  
  1. **불변**: 생성 시 한 번만 넣고 바꿀 수 없어서, 실행 중 의존이 바뀌는 실수를 막을 수 있음.  
  2. **필수 의존이 명확**: 생성자에 넣으면 "이 객체가 동작하려면 이게 꼭 필요하다"가 코드만 봐도 드러남.  
  3. **테스트**: 테스트에서 new OrderService(mockRepo) 처럼 직접 넣기 쉬움.  
  4. **순환 참조**: 생성자 주입은 스프링이 빈 만들 때 순환 참조를 **즉시** 알아차려서 에러로 알려 줌. (필드/setter는 나중에 터질 수 있음)
- **면접 답**: "생성자 주입은 의존을 불변으로 두고, 필수 의존이 드러나며, 테스트와 순환 참조 발견에 유리해서 권장합니다."

## 1. IoC (제어 역전) / DI (의존성 주입)

- **IoC**: "제어의 역전" – 객체를 **내가 new 하지 않고**, **스프링(컨테이너)**이 만들어 주고 관리함.
- **DI**: "의존성 주입" – 클래스가 필요로 하는 **다른 객체(의존)**를 **스프링이 넣어 줌**.  
  예: `Service`가 `Repository`를 쓰려면, `new Repository()` 대신 **생성자·필드**로 받음.

**왜 쓰나?**

- 객체를 한 곳에서 관리해서 **테스트·교체**가 쉬움.
- `new`를 직접 안 쓰면 **결합도**가 낮아짐.

---

## 2. Bean (빈)

- **Bean** = 스프링이 **생성·관리**하는 객체.
- 보통 **싱글톤** (앱에서 같은 타입 빈이 하나만 있음).
- 빈으로 등록하는 방법:
  - **@Component**, **@Service**, **@Controller**, **@RestController** → 클래스에 붙이면 스프링이 스캔해서 빈 등록.
  - **@Bean** → 설정 클래스의 메소드에 붙이면, 그 메소드가 반환하는 객체가 빈으로 등록됨.

---

## 3. 자주 쓰는 어노테이션 (구조)

| 어노테이션 | 붙이는 곳 | 의미 |
|------------|-----------|------|
| **@SpringBootApplication** | main 있는 클래스 | Spring Boot 앱 **시작점**. 컴포넌트 스캔 + 자동 설정 포함. |
| **@RestController** | 클래스 | 웹 요청 처리 + 응답을 **JSON** 등으로 (REST API). |
| **@Controller** | 클래스 | 웹 요청 처리 (뷰 이름 반환할 때). |
| **@Service** | 클래스 | 비즈니스 로직용 빈. (@Component와 동작 비슷) |
| **@Repository** | 클래스 | DB 접근용 빈. (JPA 등에서 사용) |
| **@Component** | 클래스 | "이 클래스는 빈으로 등록해라." |
| **@Autowired** | 생성자/필드/세터 | "이 타입의 빈을 **주입**해라." (DI) |
| **@Bean** | 메소드 | "이 메소드가 반환하는 객체를 빈으로 등록해라." |
| **@Value("${키}")** | 필드 | application.properties 값 주입. |

## 4. 웹 요청·응답 (컨트롤러)

| 어노테이션 | 용도 | 예시 |
|------------|------|------|
| **@GetMapping("/경로")** | GET 요청 처리 | `/hello` → "안녕" 반환 |
| **@PostMapping** | POST 요청 | 회원가입, 데이터 저장 |
| **@RequestParam("name")** | 쿼리 스트링 | `?name=김` → `String name` |
| **@PathVariable** | URL 경로 값 | `/users/{id}` → `Long id` |
| **@RequestBody** | 요청 본문(JSON) → 객체 | `{"name":"김"}` → DTO |

**예시 코드**

```java
@RestController
public class HelloController {

    @GetMapping("/hello")
    public String hello() {
        return "Hello, Spring!";
    }

    @GetMapping("/hello/{name}")
    public String helloName(@PathVariable String name) {
        return "Hello, " + name;
    }

    @GetMapping("/greet")
    public String greet(@RequestParam(defaultValue = "Guest") String name) {
        return "Hi, " + name;
    }

    @PostMapping("/users")
    public UserDto createUser(@RequestBody UserDto dto) {
        return dto;  // 받은 JSON 그대로 반환 (실제론 DB 저장 등)
    }
}
```

## 5. Spring MVC 흐름 (한 줄)

1. **클라이언트** → HTTP 요청 (예: GET /hello)
2. **DispatcherServlet** → 요청 받음
3. **HandlerMapping** → "이 URL은 이 컨트롤러·메소드다" 매핑
4. **Controller** → 메소드 실행, 결과 반환
5. **ViewResolver** 또는 **직접 응답**(JSON) → 클라이언트에게 응답

## 7. 예제 프로젝트 위치·실행

- **폴더**: `spring-example/`
- **실행**: `cd spring-example` → `./mvnw spring-boot:run` (또는 IDE에서 `DemoApplication` 실행)
